# Plan 8 — BI Layer: Power BI on MARTS (.pbip + pbi-cli)

**Phase:** 8 (L6) · **Complexity:** 🔴 Complex · **Status:** ⬜ Not started
**Decisions:** `DECISIONS.md` ADR-011 (Power BI / .pbip / pbi-cli), ADR-012 (RBAC least privilege).

## Goal
Surface the star schema (`MARTS`) as a trustworthy Power BI report — built in the **`.pbip`
text format** (TMDL model + PBIR JSON, version-controlled in the repo) and authored with
**`pbi-cli`**. Deliverable: a connected semantic model + a small, honest set of dashboards whose
headline numbers **reconcile to the dbt marts**.

## Locked approach (ADR-011)
- **Power BI**, **`.pbip`** format (not `.pbix` binary), built via **`pbi-cli`**.
- pbi-cli does NOT connect to Snowflake — the Snowflake→MARTS connection is a one-time
  owner-driven Desktop step.

### pbi-cli scope correction (verified at install, v1.0.6 — 2026-06-22)
The installed CLI (`pbi.exe`, package `pbi-cli-tool` 1.0.6) wraps a Power BI **MCP server**
and is **semantic-model-only**: command groups are `table/column/measure/relationship/hierarchy/
dax/security-role/calc-group/perspective/model/partition/...`. **There is NO `report`/`page`/
`visual`/`bookmark` command** — so the report VISUALS are not CLI-buildable in this version.
Division of labour, corrected:
- **Semantic model → Claude drives via pbi-cli** (relationships incl. role-playing-date inactive
  rels, DAX measures, RLS, folders). The hard/valuable part; the AE showcase.
- **Report visuals → owner builds in Desktop GUI** (drag-drop) — or hand-edit PBIR JSON (fiddly).
- Either way the `.pbip` saves model (TMDL) + report (PBIR) as **text in the repo**.
- 5 Claude skills installed by `pbi skills install`: power-bi-{dax,modeling,deployment,security,docs}.
- Workflow to attach: open the model in Desktop → `pbi connect` (auto-detects Desktop, downloads
  the MCP binary) → Claude drives modeling commands.

## Two sub-decisions — CONFIRMED 2026-06-22
1. **Read access to MARTS → new `OLIST_REPORTER` role (CONFIRMED, built & proven 2026-06-22).**
   ADR-012 gave us LOADER + TRANSFORMER only; neither is a clean BI reader, and we'd otherwise
   query MARTS as ACCOUNTADMIN. Created `OLIST_REPORTER` (USAGE on DB/WH + USAGE on MARTS schema +
   SELECT on all MARTS tables/views, incl. future; nothing else). Proven via `USE SECONDARY ROLES
   NONE; USE ROLE OLIST_REPORTER;` → reads MARTS (99,441) but **denied** on RAW.
   **Auth method (CHANGED 2026-06-23):** intended key-pair (`TYPE=SERVICE`), but the owner's Power
   BI Desktop build offers **no key-pair auth option** (GA'd only late 2025). Switched the BI user
   `OLIST_REPORTER_SVC` to **`TYPE=LEGACY_SERVICE` + password** (MFA-exempt service type). The
   **role scoping — the real least-privilege win — is unchanged**; only the auth method differs from
   the loader/transformer key-pair users. Defensible, documented PBI-GUI limitation. The unused
   `.keys/olist_reporter.*` key-pair can be removed in cleanup. _(Owner asked whether USERADMIN fits — no: USERADMIN is
   a system identity-management role with no data privileges; granting it SELECT mixes user-admin
   and data concerns. A purpose-built reader role is the least-privilege answer.)_
2. **Storage mode → Import (CONFIRMED).** Static dump → load MARTS once into Power BI's VertiPaq
   engine: fast visuals, no live warehouse cost, works offline for screenshots. DirectQuery only
   buys live freshness we don't need.

## Report scope (guarded — 2 core pages + 1 stretch)
Tie each to the ops/sales analytics the marts enable. Build **Pages 1–2 first; Page 3 only if
time allows** (scope guard).

- **Page 1 — Sales & Revenue Overview** (`fct_orders`, `fct_order_items`, `dim_dates`,
  `dim_products`, `dim_customers`): KPIs (total revenue BRL + USD/EUR via the FX columns, order
  count, AOV, items sold); revenue trend over time; revenue by product category; revenue by
  customer state; payment-type mix.
- **Page 2 — Delivery & Operations Performance** (`fct_orders`): on-time rate (`is_late`), avg
  `delivery_days` vs estimate (`delivery_vs_estimate_days`), delivery-time distribution, late-rate
  by state and by seller, review-score vs late-delivery relationship.
- **Page 3 (stretch) — Customer Segmentation** (`customer_summary`): RFM-segment distribution,
  CLV (historical) and AOV by segment, repeat vs one-time split. **Must surface the honest caveat
  on the page:** RFM here is effectively R×M (Olist ~97% one-time buyers; F scored by value).

## Modeling notes (the interesting bits)
- **Star relationships:** facts → dims on the key columns the dbt marts already expose
  (`customer_id`, `*_date_key`, product/seller keys). Verify exact columns against
  `fct_order_items` at build time.
- **Role-playing dates (the Power BI wrinkle):** `fct_orders` has **three** date FKs
  (purchase / delivered / estimated) into one `dim_dates`. Power BI allows only **one active**
  relationship between two tables → create the other two as **inactive** relationships and expose
  them via `USERELATIONSHIP` measures, OR import `dim_dates` multiple times. **Recommend** the
  inactive-relationship + `USERELATIONSHIP` pattern (one date table, explicit measures) — teach
  this as the PBI analogue of dbt's role-playing dimension.
- **Measures over columns:** revenue / AOV / on-time-rate / late-rate as explicit DAX measures
  (named, documented), not implicit aggregations.

## Honest caveats (carry into the README, Phase 9)
- RFM is R×M-dominant (F by value, not NTILE); CLV is **historical**, not predictive.
- FX conversion uses the purchase-date rate with weekend/holiday LOCF gap-fill.
- Static dump → the dashboard is a point-in-time snapshot, not a live feed (Import mode).
- pbi-cli is a young third-party tool; if it blocks us, fall back to hand-building the `.pbip`
  in Desktop and screenshotting (deliverable unchanged).

## Complexity rationale (🔴)
New, unproven tool (pbi-cli) + a new RBAC role + DAX semantic modeling + the role-playing-date
wrinkle that has no one-click answer in Power BI. Several owner-driven Desktop steps. Expect
iteration on steps 4–5.
