# Phase 0 — Project Setup & Environment

**Complexity:** ✅ Simple (single-pass, low risk) — except the owner-driven Python install.
**Status:** In progress · _Started 2026-06-16_

## Context

Before any pipeline work, the owner needs a working local Python toolchain for the
two locally-run tools: **dlt** (extract+load) and **dbt-snowflake** (transform).
This is the owner's first time with this stack, so each step is guided and verified.

State established at planning time:
- The **9 Olist CSVs are already in `data/raw/`** (verified, correct names) — the
  Kaggle download step is **done**.
- Only **Microsoft Store Python 3.13** was installed; no virtualenv existed.

Decisions confirmed:
- **Interpreter:** install **python.org Python 3.12** (broadest dbt/dlt
  compatibility; avoids Store-Python venv/path quirks). Owner installs; Claude guides + verifies.
- **One shared venv** at repo root (`.venv`, already gitignored) for dlt + dbt,
  per the existing single-`requirements.txt` decision.

Outcome: an activated, reproducible `.venv` where `dlt --version` and
`dbt --version` (with the Snowflake adapter) both work, ready for Phase 1.

---

## Steps

0. Save this plan to `plans/0.environment-setup.md`. _(this file)_
1. **Install Python 3.12 (owner-driven, Claude guides).** Download from python.org
   (3.12.x, Windows 64-bit), install with "Add python.exe to PATH" / py launcher.
   Verify: `py -3.12 --version` → `Python 3.12.x`.
2. **Create the virtualenv** on 3.12 explicitly: `py -3.12 -m venv .venv` at repo root.
3. **Install dependencies** via the venv interpreter directly (avoids PowerShell
   execution-policy issue):
   - `.venv\Scripts\python.exe -m pip install --upgrade pip`
   - `.venv\Scripts\python.exe -m pip install -r requirements.txt`
4. **Document interactive activation:** if PowerShell blocks it, run
   `Set-ExecutionPolicy -Scope Process -ExecutionPolicy RemoteSigned`, then
   `.\.venv\Scripts\Activate.ps1`.
5. **Verify the toolchain:**
   - `.venv\Scripts\dlt.exe --version`
   - `.venv\Scripts\dbt.exe --version` → must list the **snowflake** adapter
   - `.venv\Scripts\python.exe -m pip check` → no broken dependencies
6. **Pin for reproducibility:** update `requirements.txt` to pin installed top-level
   versions (e.g. `dlt[snowflake]==x.y.z`, `dbt-snowflake==x.y.z`).
7. **Confirm later-phase prerequisites (no action now):** Snowflake 180-day account
   accessible (setup is Phase 1); Astro CLI deferred to Phase 7.
8. **Update `PROGRESS.md`** — tick Phase 0 boxes; set Phase 0 to `[x]` when green.
9. **Commit** (ask owner first; no AI attribution per `CLAUDE.md`).

## Risks & fallback

- **Dependency conflict** between `dlt[snowflake]` and `dbt-snowflake` in one venv
  (both pull `snowflake-connector-python`). If `pip install`/`pip check` fails on a
  version clash, fall back to **two venvs** (`dlt/.venv`, `dbt/.venv`) and record it
  in `DECISIONS.md`. Do not force-resolve silently.
- **Store Python interference** — mitigated by always invoking `py -3.12` explicitly.

3. `.venv\Scripts\dbt.exe --version` → core version **and** the snowflake adapter
4. `.venv\Scripts\python.exe -m pip check` → "No broken requirements found."
5. `data/raw/` contains the 9 CSVs (already confirmed).
