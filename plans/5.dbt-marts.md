## Context

Phases 0–4 are complete and committed. The pipeline lands RAW (dlt) → STAGING
(10 `stg_olist__*` views) → INTERMEDIATE (6 `int_olist__*` views + 4 macros).
The hard problems are already solved in intermediate: FX gap-fill, payment
pivot (1/order), review dedup, product PT→EN enrichment, `int_olist__orders_enriched`
(order-grain hub, no fan-out, feeds `fct_orders`), and the auditable rejects table.

**Phase 5 builds the star schema** in the `MARTS` schema — the real dimensional
modeling work. Locked decisions (do not relitigate): two facts not one (ADR-007),
two-layer customer grain (ADR-006). Owner's stated learning goal: **justify each
view-vs-table materialization call** and learn surrogate keys / star schema.

## Design decisions (resolved via grill)

**D1 — Keying strategy: HYBRID.** Keep existing opaque natural keys
(`customer_id`, `product_id`, `seller_id`, `order_id`) as join keys for entity
dims — already unique/stable, and no SCD here. Generate a true surrogate only
where it adds value: `date_key` (YYYYMMDD integer) for `dim_dates`; `zip_prefix`
is the natural key for `dim_geography`. Rationale: apply surrogate keys where they
earn their place, not cargo-cult over already-unique hashes.

**D2 — Geography: CONFORMED `dim_geography`.** Keyed on `zip_prefix`, holds
city/state/lat/lng once (from `stg_olist__geolocation`). `dim_customers` and
`dim_sellers` carry their `zip_prefix` as an FK. Single source of truth;
demonstrates a conformed dimension. Accepts a mild snowflake (dim→dim). Zips absent
from geolocation = left-join nulls (signal).

**D3 — Materialization: ALL marts = `table`, no incremental.** Every dim, both
facts, and `customer_summary` materialized as tables. BI reads them repeatedly and
all upstream is views, so a mart-as-view recomputes the whole DAG per query.
Incremental skipped (static source; already demonstrated at the dlt load layer).

**D4 — `dim_dates`: full calendar years + BR holiday flag.** Span 2016-01-01 →
2018-12-31 (`dbt_utils.date_spine`, end-exclusive → pad +1 day). ISO weekday/week
(`dayofweekiso`/`weekiso`, session-independent). PK `date_key` = YYYYMMDD integer.
**Static seed** `dbt/seeds/br_holidays.csv` (2016–2018 BR national holidays incl.
Easter-derived movable: Carnival, Good Friday, Corpus Christi) → `is_holiday` /
`holiday_name`. Static seed ≠ live source (ADR-004 holds); lands in STAGING.

**D5 — Role-playing dates: 3 roles + keep timestamps.** `fct_orders` carries
`order_purchase_date_key`, `order_delivered_customer_date_key` (null when
undelivered = signal), `order_estimated_delivery_date_key`, each FK→`dim_dates`,
plus all raw timestamps. `fct_order_items` carries `order_purchase_date_key`.

**D6 — Item-grain FX: revenue only → USD + EUR.** `fct_order_items` keeps
`price_brl`, `freight_brl`, `revenue_brl`, adds `revenue_usd`, `revenue_eur` =
`brl_to(revenue, rate)` on the order's purchase date → reconciles with `fct_orders`.

**D7 — `customer_summary` monetary basis: item revenue (price + freight).**
Person grain (`customer_unique_id`). Merchant-received money (excludes card-issuer
financing). RFM via NTILE(5) quintiles (deterministic tie-break); Recency vs dataset
max order date (static); AOV; **historical** CLV (= total monetary). Builds the
deferred `rfm_bucket` + `aov` macros.

**D8 — City/state: native on dims + zip FK for coordinates.** `dim_customers`/
`dim_sellers` keep their source city/state (full coverage); carry `zip_prefix` as FK
to `dim_geography` for lat/lng. Refines D2 so city/state has no coverage gap.

## Conventions

- Location: `dbt/models/marts/` (flat). `dbt_project.yml`:
  `marts: { +materialized: table, +schema: MARTS }`; `seeds: +schema: STAGING`.
- Schema doc/tests in `dbt/models/marts/_marts__models.yml`.
- Reuse macros (`brl_to`, `order_item_revenue`, `delivery_days`, `is_late`); new
  `aov` + `rfm_bucket` in `dbt/macros/`.

## Target models (MARTS)

**Dimensions** — `dim_geography` (←stg geolocation, zip PK, 19,015), `dim_customers`
(←stg customers, customer_id PK, 99,441), `dim_sellers` (←stg sellers, 3,095),
`dim_products` (←int products_enriched, 32,951), `dim_dates` (←date_spine + holiday
seed, date_key PK, 1,096).

**Facts** — `fct_orders` (←int orders_enriched, order_id grain + 3 date keys,
99,441), `fct_order_items` (←stg order_items ⋈ orders + FX, (order_id,order_item_id)
grain, 112,650).

**Person-grain** — `customer_summary` (←fct_orders ⋈ dim_customers, customer_unique_id
grain, 96,096): frequency, monetary, aov, recency_days, first/last_order_date,
tenure_days, R/F/M (NTILE 5), rfm_segment, clv_historical.

## Build order (DAG-respecting)

`dbt seed`. 3. Macros `aov`, `rfm_bucket`. 4. Dims. 5. Facts. 6. customer_summary.
